# psv_dedup_bloom

`psv_dedup_bloom` は、大規模な PSV データから PackedSfen 重複を近似的に除去するツールです。

- 入力: PackedSfenValue 40 バイト固定長ファイル
- 重複キー: 先頭 32 バイトの PackedSfen
- 出力: 最初に出現した局面のみを残した単一ファイル
- 特徴: 1 パス、ストリーミング処理、一時ディスク不要

通常の `HashSet` で全局面を保持するとメモリが足りなくなるため、このツールは巨大なビット表を使って「見たことがある局面か」を近似判定します。

## 仕組み

各局面に対して:

1. PackedSfen 32 バイトから 2 つのハッシュ値を計算する
2. それを使ってビット表の複数位置を決める
3. まだ 0 のビットが 1 つでもあれば新規とみなして出力し、対応ビットを立てる
4. すべて 1 なら既出とみなして捨てる

この方法では、局面そのものを全部保持せずに重複判定できます。

## `fpr` とは

`fpr` は `false positive rate` の略で、偽陽性率を意味します。

このツールにおける偽陽性とは:

- 本当は新規の局面なのに
- 既出だと誤判定して捨ててしまう

ことです。

逆に、既出局面を新規と誤判定すること（偽陰性）はありません。これはブルームフィルタの設計上保証されています。

重複が検出された場合、最初に出現したレコードを保持し、以降の重複を除去します（first-wins 方式）。

`fpr` を小さくすると:

- 偽陽性は減る
- 必要メモリは増える
- 1 局面あたりに確認するビット数も増え、実行時間もやや増える

## フィルタサイズ

必要メモリは固定値ではなく、以下で自動計算されます。

- 入力レコード数の見積もり
- `fpr`

したがって、同じツールでも:

- 入力件数が多いほど大きくなる
- `fpr` を厳しくするほど大きくなる

例えば約 246 億レコードを `fpr=0.001` で処理する場合、フィルタサイズは約 40 GiB 台になります。

このサイズは入力ファイルサイズではなく、重複判定のためにメモリ上へ確保するビット表のサイズです。

### 参考値

以下は実際の大規模データでの参考値です。

- 入力ファイル総サイズ: 約 919 GiB (986 GB)
- 推定レコード数: 約 246.6 億件

| `fpr` | 推定フィルタサイズ |
|---|---:|
| `0.01` | 約 27.5 GiB |
| `0.001` | 約 41.3 GiB |
| `0.0001` | 約 55.0 GiB |

このように、`fpr` を厳しくするほど必要メモリは増えますが、10 倍厳しくしてもメモリが 10 倍になるわけではありません。

## なぜ blocked 実装にしているか

標準的な方法では、1 局面ごとに複数回のランダムメモリアクセスが発生し、フィルタが大きくなるとメモリレイテンシが支配的になります。

この実装では、ビット表を 64 byte 単位の小さなブロックに分け、

- まず 1 つのブロックを選ぶ
- そのブロック内で複数ビットを確認する

ようにしています。

これにより:

- 広いメモリ領域を何度も飛び回りにくい
- キャッシュミスを減らしやすい
- 大規模入力での処理速度が改善しやすい

という利点があります。

## メモリ不足時の挙動

Linux では `/proc/meminfo` の `MemAvailable` を見て、必要フィルタサイズが利用可能メモリの 80% を超える場合はエラーで停止します。

表示される案内:

- 必要メモリ量
- 利用可能メモリ量
- `--fpr` を緩めた場合の例
- `--force` で強制続行できること

`--force` を使うとメモリ不足警告を無視して続行できますが、swap 使用や著しい速度低下の可能性があります。

## 使用例

### 基本: ディレクトリ内の全ファイルを dedup

```bash
cargo run --release --bin psv_dedup_bloom -- \
  --input-dir ../bullet-shogi/data/DLSuisho15b \
  --pattern "*.bin" \
  --output /path/to/deduped.bin
```

### reference モード: 既存データとの差分だけを抽出

既に dedup 済みのファイルがあり、新規データから重複を除いた差分だけがほしい場合に使います。
reference ファイルは bloom filter への登録のみ行い、出力しません。

```bash
cargo run --release --bin psv_dedup_bloom -- \
  --reference existing_deduped.bin \
  --input new_data.bin \
  --output unique_new.bin
```

これにより:

- `existing_deduped.bin` の全局面を bloom に登録（出力しない）
- `new_data.bin` のうち `existing_deduped.bin` に含まれない局面だけを `unique_new.bin` に出力

既存ファイルの読み直し・再書き出しが不要なため、通常モードと比べて I/O が大幅に削減されます。

### fpr を変更する例

```bash
cargo run --release --bin psv_dedup_bloom -- \
  --input-dir ../bullet-shogi/data/DLSuisho15b \
  --pattern "*.bin" \
  --output /path/to/deduped.bin \
  --fpr 0.01
```

## オプション一覧

| オプション | 説明 | デフォルト |
|---|---|---|
| `--reference` | 参照ファイル（カンマ区切りで複数可）。bloom に登録するが出力しない | — |
| `--input` | 入力ファイル（カンマ区切りで複数可）。`--input-dir` と排他 | — |
| `--input-dir` | 入力ディレクトリ。`--pattern` と組み合わせて使用 | — |
| `--pattern` | `--input-dir` 使用時の glob パターン | `*.bin` |
| `--output` | 出力ファイルパス | — |
| `--fpr` | 偽陽性率 (0.0〜1.0) | `0.001` |
| `--max-positions` | 処理する入力の最大レコード数（0 = 全件）。試走に便利 | `0` |
| `--force` | メモリ不足でも強制続行 | off |

## 注意

- 近似重複除去なので、出力は厳密な exact dedup と一致しません
- `fpr` を小さくしすぎるとメモリ使用量が大きくなります
- 出力は単一ファイルです
- 入力ファイル自体は変更しません
- `--max-positions` は入力ファイルの処理上限であり、reference ファイルは常に全件読み込まれます
